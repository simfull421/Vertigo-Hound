Unity 6.3 LTS 기반 3D 게임을 위한 A* Pathfinding Project 도입 및 대규모 AI 전투 로직 구현에 관한 심층 연구 보고서
서론: 대규모 3D 전투 환경에서의 길찾기 및 군집 제어 패러다임 전환
현대 3D 게임 개발, 특히 다수의 인공지능(AI) 에이전트가 실시간으로 교전하는 실시간 전략 시뮬레이션(RTS), 디펜스, 대규모 난투 액션 게임을 설계할 때 가장 심각한 기술적 병목 현상은 길찾기(Pathfinding)와 군집 제어(Crowd Control)의 결합에서 발생한다. 단일 에이전트가 정적인 환경에서 목적지로 이동하는 것은 단순한 최단 거리 알고리즘으로 해결될 수 있으나, 수백 기의 아군과 적군이 유동적으로 이동하며 진형을 갖추고 교전하는 상황에서는 상황이 완전히 달라진다. 이러한 복잡한 동적 환경에서 Unity 엔진의 기본 내비게이션 시스템(NavMesh)과 내장 물리 엔진(Rigidbody, Collider)에 전적으로 의존하게 되면, 에이전트 간의 교착 상태(Deadlock), 과도한 물리 연산으로 인한 프레임 저하, 그리고 목적지 주변에서 유닛들이 끊임없이 떨리는 진동(Jittering) 현상이 필연적으로 발생하게 된다.   

이러한 다차원적인 기술적 난제를 단일 아키텍처 내에서 우아하게 해결할 수 있는 가장 강력하고 검증된 대안이 바로 Aron Granberg가 개발한 A* Pathfinding Project이다. 유니티 생태계에서 오랜 기간 표준적인 위치를 차지해 온 이 에셋은, 단순한 길찾기 알고리즘의 집합을 넘어 고도의 군집 시뮬레이션과 공간 쿼리 최적화를 제공하는 포괄적인 프레임워크로 진화하였다. 특히 최신 메이저 업데이트인 버전 5.x는 Unity의 최신 고성능 아키텍처인 Burst Compiler, C# Job System, 그리고 Entity Component System(ECS)을 전면적으로 채택하여 기존 버전 대비 비약적인 연산 성능 향상을 이룩하였다.   

본 연구 보고서는 Unity 6.3 LTS 환경에서 3D 전투 게임을 개발하는 상황을 전제로, A* Pathfinding Project를 활용하여 복잡다단한 AI 간의 전투 로직을 어떻게 효율적이고 안정적으로 구현할 수 있는지에 대한 심층적인 통찰과 구조적 가이드라인을 제공한다. 도입부의 환경 설정부터 시작하여, 3D 지형을 위한 그래프 생성, ECS 기반의 초고속 이동 스크립트 활용, 로컬 회피(RVO) 알고리즘을 통한 부드러운 군집 제어, 그리고 실제 교전 상태(접근, 정지, 타격, 회전)에서의 세밀한 파라미터 튜닝에 이르기까지 모든 과정에 대한 기술적 분석을 전개할 것이다.

※ 본 프로젝트의 적 AI는 AIPath/RichAI 기반 IAstarAI 이동을 사용하며, FollowerEntity 관련 내용은 선택 옵션 참고용입니다.

1. Unity 6.3 LTS 환경 구성 및 아키텍처 호환성 최적화
새로운 시스템을 도입함에 있어 가장 먼저 선행되어야 하는 것은 개발 환경과의 완벽한 호환성을 확보하고, 의존성 패키지 충돌을 예방하는 것이다. A* Pathfinding Project 버전 5는 그 자체로 거대한 연산 엔진이므로, Unity 6.3 LTS의 새로운 컴파일 파이프라인과 패키지 관리 시스템에 정확히 맞물리도록 구성되어야 한다.

1.1 필수 의존성 패키지 설치 및 ECS 프레임워크 통합
버전 5로 도약하면서 이 시스템은 내부 연산의 상당 부분을 Unity의 Data-Oriented Technology Stack (DOTS) 기술에 의존하게 되었다. 따라서 패키지가 정상적으로 작동하고 최고의 퍼포먼스를 내기 위해서는 패키지 관리자(Package Manager)를 통한 추가적인 모듈 설치가 필수적이다. 기본적으로 Mathematics, Collections, 그리고 Burst 패키지가 요구된다. 특히 Burst 컴파일러는 메모리 할당을 최소화하고 벡터화된 기계어 코드를 생성함으로써 그리드 그래프의 스캔 속도를 최대 3배 이상, 군집 회피 연산 속도를 기하급수적으로 향상시키는 핵심 동력이다.   

대규모 AI 전투 로직을 기획하고 있다면, 선택적 의존성(Optional Dependency)으로 분류되는 Entities 패키지의 설치를 강력히 권장한다. 기존의 AIPath 스크립트를 대체하는 고성능 스크립트인 FollowerEntity를 구동하기 위해서는 Entities 패키지가 반드시 프로젝트에 포함되어 있어야 하기 때문이다. Unity 6.3 LTS 환경에서 Entities 패키지(버전 1.20 이상)를 설치할 때, 기존 레거시 프로젝트에서 업그레이드하는 경우 간혹 Unity.Mathematics, Burst, Jobs 등의 네임스페이스를 인식하지 못하여 수천 개의 컴파일러 오류가 발생하는 현상이 보고되기도 한다. 이러한 현상은 패키지 간의 종속성 트리가 꼬였을 때 발생하므로, 라이브러리(Library) 폴더를 완전히 삭제하고 프로젝트를 재시작하여 패키지 캐시를 강제로 갱신하는 방식으로 해결할 수 있다. 만약 Entities 패키지를 설치하지 않은 상태에서 FollowerEntity 스크립트가 포함된 데모 씬을 실행하거나 컴포넌트를 부착하려 시도하면, 내부적으로 참조 누락이 발생하여 스크립트가 정상적으로 직렬화되지 않으니 주의가 필요하다.   

1.2 어셈블리 참조 오류(Assembly Reference Error)의 식별 및 해결
Unity 6 이상의 환경에 이 패키지를 임포트할 때 개발자들이 가장 빈번하게 직면하는 장애물은 어셈블리 참조와 관련된 컴파일 오류이다. 구체적으로는 Assembly 'Library/ScriptAssemblies/AstarPathfindingProjectEditor.dll' will not be loaded due to errors: Unable to resolve reference 'AstarPathfindingProject'와 같은 형태의 오류 메시지가 Unity 콘솔을 가득 채우게 된다.   

이 오류는 패키지 자체의 결함이 아니라, 프로젝트 내부의 코드 조직화 방식, 구체적으로는 어셈블리 정의 파일(.asmdef) 시스템의 캡슐화 특성에서 기인한다. Unity는 컴파일 시간을 단축하고 모듈 간의 결합도를 낮추기 위해 .asmdef 파일을 사용하여 코드를 여러 개의 독립적인 어셈블리로 분리하는 것을 권장한다. 만약 개발 중인 3D 게임 프로젝트의 스크립트들이 특정 .asmdef 파일의 지배를 받고 있다면, 해당 구역 내에서는 명시적으로 권한이 부여되지 않은 외부 어셈블리(이 경우 A* Pathfinding Project의 핵심 스크립트들)의 코드를 열람할 수 없게 된다.   

이러한 상황에서는 스크립트 상단에 using Pathfinding; 이라는 지시문을 선언하더라도, Visual Studio 등의 통합 개발 환경(IDE)과 Unity 컴파일러는 해당 네임스페이스를 찾을 수 없다는 치명적 오류를 반환한다. 이 문제를 해결하고 참조 무결성을 복구하기 위한 절차는 다음과 같다.   

해결 단계	세부 작업 지침 및 메커니즘 설명
문제 영역 식별	
AI 제어 스크립트나 커스텀 이동 로직이 작성된 디렉토리 내에 존재하는 사용자의 커스텀 .asmdef 파일을 찾아 선택한다.

참조 슬롯 활성화	
Unity Inspector 창에서 해당 어셈블리 정의의 Assembly Definition References 섹션을 펼치고 '+' 아이콘을 클릭하여 새로운 참조 슬롯을 생성한다.

어셈블리 연결	
나타난 슬롯의 우측 원형 버튼을 클릭하여 프로젝트 내에 존재하는 AstarPathfindingProject 어셈블리를 검색하여 할당한다. 에디터 전용 확장 스크립트를 작성하는 경우 AstarPathfindingProjectEditor 어셈블리도 함께 추가해야 한다.

적용 및 컴파일	
하단의 'Apply' 버튼을 눌러 변경 사항을 저장한다. 이 순간 Unity는 어셈블리 종속성 그래프를 재구성하며 컴파일을 수행한다.

검증	
IDE로 돌아가 using Pathfinding; 선언의 붉은 밑줄이 사라지고 코드 자동 완성(IntelliSense) 기능이 정상적으로 A* 패키지의 클래스들을 인식하는지 확인한다.

  
만약 프로젝트 내에 .asmdef 파일을 전혀 사용하지 않는 단일 어셈블리 구조(Assembly-CSharp.dll)임에도 불구하고 유사한 클래스 인식 불량이나 Missing Reference 오류가 발생한다면, 이는 Unity 버전 업그레이드 과정에서 발생한 직렬화 데이터의 오염이 원인일 가능성이 크다. 이 경우에는 프로젝트의 파일 시스템으로 직접 접근하여 AstarPathfindingProject 폴더 전체를 수동으로 삭제한 뒤, 에디터를 재시작하고 에셋 패키지를 다시 처음부터 임포트하는 클린 인스톨(Clean Install) 방식을 취해야만 내부 캐시가 정상화된다.   

1.3 Unity 6.3 LTS API 변경 사항 대응 (Obsolete Warnings)
Unity 엔진 자체의 API가 발전함에 따라, 서드파티 에셋 내부에 작성된 일부 구형 API 호출이 경고를 발생시킬 수 있다. 대표적으로 Unity 6.3 LTS 환경에서 A* Pathfinding Project의 이동 통합 스크립트(또는 이를 활용하는 Behavior Designer와 같은 연계 시스템)를 컴파일할 때, IAstarAI.canMove 속성이 더 이상 사용되지 않음(Obsolete)을 알리는 경고가 출력된다. 컴파일러는 이 속성이 simulateMovement라는 새로운 명칭으로 대체되었음을 안내한다. 비록 이러한 Obsolete 경고가 당장의 게임 실행을 가로막는 치명적인 에러(Error)는 아니지만, 장기적인 코드 유지보수와 후속 Unity 버전(예: Unity 7)에서의 호환성을 담보하기 위해서는 커스텀 AI 스크립트 내부에서 에이전트의 이동 가능 여부를 제어할 때 canMove 대신 simulateMovement 프로퍼티를 명시적으로 사용하는 방향으로 코드를 리팩토링하는 것이 바람직하다.   

2. 3D 전투 환경을 위한 최적의 공간 분할 기법: Recast Graph 아키텍처
AI 에이전트들이 3D 공간 내에서 최적의 경로를 찾기 위해서는, 연속적인 3D 물리 공간을 수학적으로 계산 가능한 불연속적인 노드(Node)의 집합으로 변환하는 과정이 선행되어야 한다. 이를 '그래프 생성(Graph Generation)'이라고 한다. A* Pathfinding Project는 2D 타일 기반 게임에 적합한 Grid Graph, 웨이포인트 기반의 Point Graph 등 다양한 그래프 타입을 지원하지만, 높낮이가 존재하는 지형, 다층 구조의 건물, 그리고 계단과 경사로가 복잡하게 얽힌 3D 전투 환경에서는 Recast Graph를 채택하는 것이 기술적으로 가장 합당한 선택이다.   

2.1 Recast Graph의 작동 원리와 압도적 우위
Unity의 기본 내장 NavMesh 시스템 역시 훌륭한 기능을 제공하지만, 소규모 정적 환경에 최적화되어 있다는 태생적 한계를 지닌다. 반면 A* 패키지의 Recast Graph는 훨씬 더 정교하고 유연한 제어 권한을 제공한다. Recast 프로세스는 본질적으로 씬 내에 존재하는 모든 지정된 메쉬(Mesh)와 콜라이더(Collider) 기하학 구조를 정밀한 육면체 형태의 복셀(Voxel) 단위로 잘게 쪼개는 복셀화(Voxelization) 과정으로 시작된다. 이후 에이전트가 통과할 수 없는 빈 공간과 가파른 경사면을 걸러내고, 최종적으로 에이전트의 반경과 높이를 고려하여 이동 가능한 표면을 삼각형 폴리곤의 집합(Triangle Mesh)으로 재구성한다.   

이 방식은 수많은 장점을 수반한다. 첫째, 지형의 구멍(Terrain Holes)이나 속이 꽉 찬 솔리드 콜라이더(Solid Colliders)를 완벽하게 인식하여 메쉬가 중첩된 복잡한 지형에서도 에이전트가 바닥으로 추락하는 현상을 원천적으로 차단한다. 둘째, 버전 5.x에 이르러 Burst 컴파일러와 C# Job System이 결합되면서 대형 맵을 스캔하는 속도가 과거와 비교할 수 없을 정도로 빨라졌다. 셋째, 다수의 타일(Tile) 구조로 그래프를 분할 관리하므로, 게임 실행 중 맵의 일부분이 변형되더라도 전체를 재계산할 필요 없이 변경된 타일 구역만 독립적으로 비동기 업데이트를 수행할 수 있어 대규모 동적 전장 구현에 유리하다.   

2.2 동적 표면 태그(Tags) 및 이동 비용(Penalties)을 통한 전술 구역 설계
전투 로직 설계에 있어 단순히 "A에서 B로 이동할 수 있는가?"를 넘어, "어떤 경로가 전술적으로 더 안전하고 효율적인가?"를 AI 스스로 판단하게 만드는 것이 핵심이다. 예컨대 늪지대, 얕은 물, 불길이 치솟는 장판 지역, 혹은 독가스가 살포된 구역 등은 물리적으로 이동은 가능하지만 통과할 때 패널티(이동 속도 감소 또는 체력 지속 감소)를 받게 된다. 뛰어난 AI라면 목적지까지 거리가 조금 멀어지더라도 이러한 위험 구역을 우회하는 경로를 우선적으로 탐색해야 한다.

이를 구현하기 위해 3D 레벨 디자이너는 맵 상의 특정 오브젝트나 프리팹에 RecastNavmeshModifier 컴포넌트를 부착하여 표면의 특성을 세밀하게 조율할 수 있다.   

표면 수정 모드 (Surface Modification Mode)	기술적 특성 및 전투 설계 적용 방안
UnwalkableSurface	
해당 메쉬의 모든 표면을 즉각적으로 이동 불가 영역으로 지정한다. 투명한 절벽 경계선이나 강제 접근 금지 구역을 설정할 때 유용하다.

WalkableSurface	
기본 동작 모드로, 장애물이 아닌 일반적인 도보 가능 지형으로 간주한다.

WalkableSurfaceWithSeam	
해당 메쉬 표면을 걷기 가능한 구역으로 지정하되, 인접한 다른 메쉬들과 연결되는 부위에 물리적인 내비메시 절취선(Seam)을 강제로 생성한다. 추후 이 특정 구역만 그래프에서 제거하거나 업데이트할 때 절취선 덕분에 계산 부하가 최소화된다.

WalkableSurfaceWithTag	
이동 가능 구역으로 설정함과 동시에, 해당 구역 내의 모든 삼각형 노드에 특정한 정수형 '태그(Tag)'를 부여한다. 이 태그는 에이전트의 이동 비용을 계산하는 척도가 된다.

  
태그 시스템의 진가는 에이전트에 부착된 Seeker 컴포넌트 내의 이동 비용(Traversal Costs) 설정과 결합될 때 발휘된다. 예를 들어, '독가스 지대'에 해당하는 태그 1번 표면에 대해 기본 이동 비용(보통 1000) 외에 추가로 50000의 막대한 페널티 코스트를 부여해둔다. AI가 경로를 탐색할 때, 내부의 A* 알고리즘 연산은 노드의 누적 비용(G Cost)과 발견적 추정치(H Cost)를 합산하여 최적 경로를 도출하는데, 이때 엄청난 페널티 코스트가 부여된 독가스 지대 노드를 밟는 것보다 맵을 멀리 우회하여 돌아가는 노드들의 합산 비용이 더 작다고 판단하게 된다. 결과적으로 AI는 마법 장판이나 늪지대를 스스로 인지하고 회피하는 고도의 지능적 기동을 선보이게 된다.   

2.3 Off-mesh Links를 통한 입체적 기동과 장애물 극복
Recast Graph의 또 다른 강력한 기능은 분리된 공간 사이를 논리적으로 연결하는 Off-mesh links이다. 공성전 게임에서 성벽 아래의 적이 성벽 위의 목표로 이동해야 할 때, 직접 걸어 올라갈 계단이 없다면 내비메시는 이 두 구역을 단절된 공간으로 인식하여 오류를 반환한다. 그러나 사다리, 점프 패드, 혹은 바리케이드를 뛰어넘을 수 있는 특정 지점에 Off-mesh link를 수동으로 배치하면, 그래프 상에는 두 지점을 잇는 가상의 엣지(Edge)가 생성된다.   

AI가 탐색한 경로 상에 이 링크가 포함되어 있다면, 에이전트가 링크 시작점에 도달하는 순간 일반적인 보행 로직을 일시 중단하고 onTraverseOffMeshLink 이벤트를 발동시킨다. 개발자는 이 이벤트를 구독하여 AI가 포물선을 그리며 점프하는 애니메이션을 재생하거나, 후크샷을 던지는 모션을 취한 뒤 도착 지점으로 트랜스폼을 이동시키는 커스텀 애니메이션 코드를 작성함으로써 매우 역동적인 3D 입체 기동 전투를 연출할 수 있다. 만약 앞을 가로막는 바리케이드가 Off-mesh link의 주체라면, FollowerEntity.GetRemainingPath() 메서드를 사용하여 전방의 경로 버퍼를 미리 읽어들인 뒤, 해당 링크가 도달 불가능한 상태라면 바리케이드 자체를 우선 타겟으로 설정하여 파괴 공격을 가하도록 유도하는 치밀한 공성 로직 설계도 가능하다.   

3. 대규모 엔티티 병렬 처리의 핵심: FollowerEntity 프레임워크
A* Pathfinding Project의 초기 버전부터 오랜 기간 AI 이동의 중추적인 역할을 담당해 온 클래스는 AIPath와 RichAI였다. 이들은 직관적이고 훌륭한 기능을 제공하지만, 본질적으로 Unity의 단일 메인 스레드 상에서 동작하는 MonoBehaviour 기반 스크립트이다. 따라서 화면에 수십 기의 에이전트만 존재할 때는 문제가 없으나, 삼국지나 토탈워 시리즈처럼 수백에서 수천 기의 AI가 동시에 독립적인 경로를 추적하고 충돌을 회피하며 타겟을 추적하는 초대규모 전투 환경에서는 스크립트의 Update 루프 자체가 엄청난 병목을 일으켜 치명적인 프레임 드랍(FPS Drop)을 초래한다.   

이러한 성능의 벽을 허물기 위해 버전 5.x에서 새롭게 창안된 아키텍처가 바로 FollowerEntity이다. 이 스크립트는 겉보기에는 기존의 스크립트처럼 인스펙터에 부착하는 단순한 컴포넌트처럼 보이지만, 그 내부의 설계 철학과 작동 원리는 Unity의 차세대 기술인 ECS(Entity Component System) 패러다임에 뿌리를 두고 있다.   

3.1 FollowerEntity의 구조적 메커니즘과 멀티스레딩
FollowerEntity를 사용하는 개발자는 복잡한 DOTS 코딩을 학습할 필요가 없다. 게임 오브젝트에 FollowerEntity 컴포넌트를 부착하고 실행하는 순간, 이 스크립트는 백그라운드에서 자신과 1:1로 대응하는 보이지 않는 고성능 '엔티티(Entity)'를 생성한다. 그리고 이 엔티티의 메모리 청크(Chunk) 내부에 LocalTransform(위치 및 회전), MovementState(현재 이동 속도 및 궤적), DestinationPoint(최종 타겟 좌표), ResolvedMovement(물리적 보정이 완료된 이동 벡터) 등 오직 연산에만 필요한 순수한 데이터 구조체 컴포넌트들을 삽입한다.   

매 프레임마다 메인 스레드의 부하를 없애기 위해 다음과 같은 순환 동기화 파이프라인이 가동된다.

데이터 입력 동기화: SyncTransformsToEntitiesSystem이라는 ECS 시스템이 메인 스레드에서 작동하며, 씬에 존재하는 GameObject들의 현재 위치 좌표를 백그라운드의 엔티티들에게 일괄적으로 복사하여 전달한다.   

병렬 연산 (Job System): 수많은 엔티티들의 데이터가 캐시 적중률(Cache Hit Ratio)이 극대화된 선형 메모리에 배열되어 있으므로, Burst로 컴파일된 C# Job System이 이 데이터를 받아 수백 개의 CPU 스레드로 잘게 쪼개어 경로 추적 속도, 가속도, 조향력을 동시에 병렬 계산한다.

데이터 출력 반영: 계산이 완료된 결과값(ResolvedMovement)은 다시 GameObject의 트랜스폼으로 전달되어 화면상의 모델이 매끄럽게 움직이게 된다.

이러한 철저한 데이터와 연산의 분리 구조 덕분에, 개발자는 기존의 객체 지향적(OOP) 방식의 편리한 개발 경험을 그대로 유지하면서도 하드웨어의 다중 코어 성능을 100% 한계까지 쥐어짜내는 극강의 최적화 효율을 달성할 수 있다.   

3.2 Time Scaling과 거대 군단 성능 관리
대규모 난투 게임에서는 전투의 긴박감을 위해 전체 게임의 진행 속도(Time Scale)를 높이거나 낮추는 슬로우 모션 연출이 자주 사용된다. 일반적인 코루틴이나 Update 기반의 이동 로직은 Time Scale이 비정상적으로 커지면 프레임 간의 이동 거리 간격이 넓어져 에이전트가 벽을 뚫고 지나가거나 내비메시 외부로 이탈하는 치명적 버그를 낳는다. 그러나 FollowerEntity는 타임 스케일이 1.0을 초과할 경우 내부적으로 시뮬레이션 틱(Simulation Step)을 여러 번 반복 수행하도록 자동 분기 처리되어, 아무리 빠른 배속으로 게임이 진행되어도 에이전트가 항상 내비메시 표면에 정밀하게 밀착(Clamping)된 채 궤도를 이탈하지 않는 안정성을 자랑한다.   

다만 주의할 점은, ECS 기반 아키텍처의 엄격한 규칙 상, 시스템이 엔티티 목록을 순회하며 연산을 수행하고 있는 도중에 컴포넌트를 임의로 삭제하거나 추가하는 '구조적 변경(Structural Change)'을 가하게 되면 InvalidOperationException: Structural changes are not allowed while iterating over entities.라는 예외 오류를 뿜어내며 크래시가 발생할 수 있다는 것이다. 따라서 외부 이동 시스템 패키지(예: Opsive Movement Pack 등)와 연동하여 커스텀 행동 로직을 작성할 때는 컴포넌트 조작을 즉시 실행하지 않고 EntityCommandBuffer를 통해 지연 처리(Deferred Execution) 하도록 설계해야 함을 명심해야 한다.   

4. 고도화된 군중 제어 및 충돌 방지: RVO 알고리즘의 3D 구현
AI들이 타겟을 향해 이동하는 로직을 구축한 후 다수의 유닛을 생성해보면 즉각적인 문제에 봉착한다. 수십 명의 전사들이 하나의 좁은 문턱을 통과하거나, 동일한 타겟 몬스터를 향해 일제히 돌진할 때 유닛들이 서로 얽혀서 오도가도 못하거나 사방으로 튕겨져 나가는 현상이다. 이는 다수의 오브젝트가 한 점으로 모일 때 겹침(Penetration)을 해결하려는 Unity의 물리 엔진(Rigidbody Physics)의 반발력 연산이 경로 탐색 알고리즘이 지시하는 방향타 연산과 정면으로 충돌하기 때문이다.   

3D 전투 환경에서 이러한 군중 병목(Crowd Bottleneck) 현상을 억제하고 액체의 흐름처럼 부드러운 우회 기동을 만들어내기 위해서는 유니티 물리 콜라이더의 충돌 연산을 과감하게 배제하고, A* Pathfinding Project가 자랑하는 로컬 회피 시스템인 RVO (Reciprocal Velocity Obstacles) 알고리즘에 이동과 회피의 전권을 일임해야 한다.   

4.1 RVO 시뮬레이션의 수학적 원리와 ORCA
RVO 시스템은 각 에이전트를 물리적인 질량을 가진 물체가 아니라, 주변 에이전트들의 '현재 위치'와 '이동 속도 벡터'를 실시간으로 분석하여 미래에 발생할 충돌 확률을 계산하는 추상적인 수학적 실체로 취급한다. 구체적으로는 ORCA (Optimal Reciprocal Collision Avoidance) 알고리즘을 변형하여 사용하는데, 이는 두 에이전트가 마주보며 다가올 때 충돌을 피하기 위해 필요한 최소한의 속도 변화량(Velocity Delta)을 계산하고, 이 책임을 두 에이전트가 반분(Reciprocal)하여 각자의 속도 벡터를 미세하게 꺾음으로써 부드럽게 비껴가게 만드는 방식이다.   

물리 엔진에 의존하지 않고 순수 연산으로 처리되기 때문에, 유닛들이 조금씩 겹치더라도 엔진이 강제적인 튕겨냄을 유발하지 않아 진동 현상이 사라지며, 수백 명의 군중이 밀집한 상태에서도 압도적으로 적은 CPU 자원으로 부드러운 유체 흐름과 같은 군집 이동을 시뮬레이션할 수 있다.

4.2 RVOSimulator 글로벌 매니저 설정
RVO 시스템이 가동되기 위해서는 씬(Scene) 전체를 아우르며 모든 에이전트들의 데이터를 취합하고 연산을 조율하는 중앙 관제탑인 RVOSimulator 컴포넌트가 씬 내에 단 하나만, 필연적으로 존재해야 한다. 이 매니저 객체에 어떤 파라미터를 세팅하느냐에 따라 군중 회피 연산의 퍼포먼스와 정밀도가 크게 좌우된다.   

파라미터 세부 설정 항목	최적화 기준 및 설정 근거
Desired FPS (목표 주사율)	
렌더링 프레임(보통 60 FPS)과 시뮬레이션 틱 레이트를 분리한다. 회피 연산은 시각적 갱신만큼 빈번할 필요가 없으므로 10 ~ 30 사이로 설정하여 막대한 CPU 자원 절감 효과를 거둔다.

Worker Threads (스레드 분배)	
다중 코어 프로세서를 적극 활용하기 위해 Automatic High Priority 모드를 선택한다. 이는 메인 스레드에 가해지는 부담을 덜어주어 수천 단위의 에이전트 운용을 가능케 한다.

Double Buffering (이중 버퍼링)	
반드시 활성화한다. 백그라운드 스레드에서 복잡한 회피 벡터를 계산하는 동안, 에이전트는 직전 프레임에 계산 완료된(더블 버퍼링된) 결과값을 읽어와 이동하므로 연산 지연에 따른 마이크로 스터터링(Stuttering)을 방지한다.

Interpolation (선형 보간)	
낮은 Desired FPS (예: 15) 환경에서 각 에이전트의 이동이 뚝뚝 끊겨 보이는 것을 막기 위해 두 시뮬레이션 스텝 사이의 위치 좌표를 시각적으로 부드럽게 보간(Lerp)하여 렌더링 프레임에 맞춘다.

Navmesh Obstacles	
그래프의 외곽 경계선이나 낭떠러지 바깥으로 유닛들이 서로를 강하게 밀어내 떨어뜨리는 것을 막으려면 useNavmeshAsObstacle을 체크해야 한다. 하지만 이 기능은 상당한 연산 오버헤드를 유발하므로 대규모 군단 전투 시에는 끄는 것이 퍼포먼스 상 이득일 수 있다. 이럴 땐 대안으로 외곽 영역에 별도의 RVONavmesh 컴포넌트를 배치하여 그래프 테두리를 고정된 RVO 장애물로 사전 치환하는 것이 훨씬 가볍고 효율적이다.

  
4.3 물리 엔진 충돌 레이어 분리 (Physics Layer Segregation)
가장 빈번하게 저지르는 실수는 RVO 시스템을 세팅해놓고서 에이전트들의 프리팹에 여전히 Rigidbody와 형태를 갖춘 Capsule Collider를 남겨둔 채 Unity 물리 시스템의 상호 충돌을 방치하는 것이다. 두 시스템이 동시에 개입하면 에이전트는 RVO가 지시하는 부드러운 우회 궤적과 물리 엔진이 강제하는 단단한 반발력 사이에서 끊임없이 모순된 명령을 받아 화면상에서 심하게 진동(Jittering)하게 된다.   

이를 완벽하게 분리하기 위해서는 모든 AI 에이전트 프리팹의 레이어를 특정한 식별 레이어(예: "AI_Units")로 일괄 지정해야 한다. 그 후 Unity의 상단 메뉴인 Edit > Project Settings > Physics로 진입하여 Layer Collision Matrix 상에서 "AI_Units" 레이어와 "AI_Units" 레이어 자신이 서로 교차하는 체크박스를 해제하여 동족 간의 물리적 충돌 연산을 완전히 무효화시켜야 한다. 이렇게 조치하면 유닛끼리 부딪혔을 때 서로를 통과하는 것처럼 보이게 되지만, 바로 그 순간 RVO 시스템의 가상 콜라이더(Agent Cylinder Shape)가 개입하여 수학적으로 완벽하게 서로를 밀어내고 비껴가게 만들어 충돌을 관리한다.   

5. 피아 식별 및 지능적 군집 전투 로직 설계 심화
군집 이동 환경 구축이 완료되었다면, 본격적으로 "아군과 적군이 조우하여 진형을 유지하고 싸우는" 전투 상태 머신(State Machine)을 통합해야 한다. 대규모 전투의 가장 큰 과제는 단순히 이동하는 것을 넘어, **'같은 팀끼리는 진로를 양보하며 유연하게 비켜주되, 적군을 향해서는 길을 비켜주지 않고 전면적으로 돌진하여 무기를 휘둘러야 한다'**는 고도의 지능적 집단 행동을 구현하는 것이다.

5.1 비트마스크(Bitmask)를 활용한 팀 기반 회피 계층화
위에서 언급한 팀 별 행동 양식의 차별화는 각 에이전트에 부여된 Collides With 비트마스크(Bitmask) 속성을 통해 매우 정교하게 조율할 수 있다. 프로그래밍 관점에서 비트마스크 연산은 단일 32비트 정수 하나만으로 32개의 상태(이 경우 레이어) 온/오프 여부를 초고속으로 판별할 수 있어 조건문 순회보다 압도적으로 메모리와 캐시 효율이 높은 기법이다.   

Layer Mask=(1≪L

예를 들어, 아군 진영을 레이어 1번(1 << 1 즉 00010)으로 정의하고, 적군 진영을 레이어 2번(1 << 2 즉 00100)으로 정의했다고 가정하자.
아군 AI 에이전트의 RVO 설정에서 Collides With 마스크 값을 레이어 1번만 포함하도록 구성한다. 이렇게 설정하면 아군 AI가 전진할 때 레이어 1에 속한 다른 아군 개체를 인지하면 ORCA 알고리즘에 의해 서로 진로를 미세하게 수정하며 어깨를 피해 지나간다. 반면 적군(레이어 2)과 마주쳤을 때는 적군이 회피 대상 레이어 마스크에 포함되지 않으므로, 적을 '존재하지 않는 가상의 공간' 혹은 '회피할 필요가 없는 빈약한 대상'으로 간주하고 전혀 속도를 줄이거나 방향을 꺾지 않고 직선으로 격돌하게 된다. 이 단순하면서도 강력한 비트 연산을 통해 난전 속에서도 피아 식별에 기반한 자연스러운 돌진과 교착 상태 묘사가 가능해진다.   

5.2 우선순위(Priority) 동적 조정을 통한 진형 붕괴 방지
적군과의 교전이 시작되어 검을 휘두르는 공격 애니메이션이 재생되는 상태(Attacking State)에 돌입한 최전선 유닛이 있다고 가정하자. 이때 후방에서 쏟아져 들어오는 아군 지원 부대 유닛들이 회피 로직을 무시하고 뒤에서 무턱대고 밀어버린다면, 최전선 유닛이 적진 한가운데로 튕겨 들어가거나 조준점이 틀어져 타격 판정이 어긋나는 어색한 상황이 발생하게 된다. 즉, "멈춰서 싸우고 있는 유닛"은 뒤에서 다가오는 "이동 중인 유닛"에 의해 밀려나서는 안 되며, 오히려 굳건한 바위처럼 버티고 이동 중인 유닛이 그 주변을 돌아가도록 만들어야 한다.

이는 에이전트의 '우선순위(Priority)' 파라미터를 동적으로 변경함으로써 우아하게 해결된다. RVO 알고리즘에서 두 유닛(A와 B)이 충돌 위험에 처했을 때, 서로 책임을 나누어 가지는 회피 강도(Avoidance Strength)의 비율은 각자의 우선순위 수치에 의해 다음과 같은 방정식으로 결정된다 :   

회피 분담률 (Avoidance Strength of A)= 
Priority of A+Priority of B
Priority of B
​
 
개발자는 AI의 상태가 '이동(Moving)'일 때는 priority 값을 기본값인 0.5로 유지하다가, 상태가 '공격(Attacking)' 혹은 '대기(Idle)'로 전환되는 순간 이 값을 1.0으로 끌어올리도록 스크립트를 작성한다. 이렇게 조치하면, 뒤에서 다가오는 우선순위 0.5의 아군 유닛은 우선순위 1.0의 교전 중인 유닛을 만났을 때 회피 연산 분담률을 전적으로 자신이 짊어지게 된다. 결과적으로 교전 중인 유닛은 묵묵히 제자리를 지키며 전투에 임하고, 후속 부대는 마치 커다란 바위를 피해 물결이 갈라지듯 교전 유닛의 양옆으로 우회하여 적군을 향해 진형을 넓게 펴며 포위하는 웅장한 진형 전개를 보여주게 된다.   

5.3 목표 접근 완료와 타격 상태의 정지 로직 (Stopping and Locking)
AI가 무기를 휘두를 수 있는 사거리(Stop Distance) 내에 진입했을 때, 목표를 잃지 않고 안정적으로 타격을 가하려면 이동을 완전히 멈추어야 한다. 초보 개발자들이 흔히 하는 실수는 타겟에 도달했을 때 FollowerEntity의 목적지(ai.destination)를 자기 자신의 현재 좌표로 덮어쓰거나, 최대 속도를 강제로 0으로 만들어버리는 것이다. 그러나 이 방식은 목적지 주변에서 끊임없이 새로운 제자리 경로를 탐색하게 만들어 연산 자원을 극심하게 낭비하며, 외부 힘에 의해 조금만 밀려나도 억지로 원래 위치로 복귀하려는 비정상적인 거동을 유발한다.   

가장 정석적이고 안전한 제어 방법은 불리언(Boolean) 플래그를 통한 상태 잠금이다.   

이동 일시 정지: 사거리 내 진입을 감지하면 즉시 followerEntity.isStopped = true를 호출한다. 이 속성은 에이전트 내부적으로 경로 계산과 목적지 정보는 그대로 보존한 채, 바퀴의 동력만을 일시적으로 끊어 멈추게 만든다.   

공간 고정 (Locking): 정지한 상태에서도 여전히 다른 RVO 유닛에 의해 밀려날 가능성이 있다. 이를 차단하기 위해 RVOController.locked = true (또는 ECS rvoSettings 구조체 내의 locked 속성) 플래그를 활성화시킨다. 더 스마트한 접근법으로는, 애초에 컴포넌트 인스펙터 상에서 Lock When Not Moving 토글을 켜두는 것이다. 이 기능이 켜져 있으면 에이전트의 목표 속도가 0에 수렴하는 순간 자동으로 위치가 고정(Lock)되어 외부의 거대한 무리로부터 밀림을 방지하는 단단한 초석 역할을 해준다. 이 전략은 특히 좁은 길목이나 다리 위에서 적의 진군을 틀어막는 방어형 탱커(바리케이드) 유닛을 구현할 때 매우 탁월한 효과를 발휘한다.   

5.4 정지 상태에서의 제자리 회전 제어와 Jitter(진동) 문제 해결
타겟에 도달하여 발이 묶인 상태(isStopped = true)라 하더라도, 적이 주변을 빙글빙글 맴돌며 이동한다면 AI는 이동하지 않고도 몸통만 회전하여 항상 타겟을 정면으로 주시(Facing)해야 한다. 기존의 단순한 transform.LookAt() 함수를 호출하는 것은 A* 에셋 내부의 엔티티 회전 동기화 로직과 충돌하여 심하게 모델이 비틀리는 부작용을 낳는다.

FollowerEntity 시스템에는 오직 이러한 상황을 위해 특별히 설계된 SetDestination 함수의 오버로드(Overload) 메서드가 준비되어 있다.   

C#
// 매 프레임 타겟을 향하는 방향 벡터를 정규화하여 계산
Vector3 directionToTarget = (target.position - transform.position).normalized;

// 목적지 좌표는 자신의 현재 위치로 고정하되, 바라봐야 할 지향 방향(Facing Direction) 매개변수를 함께 전달
followerEntity.SetDestination(transform.position, directionToTarget);
이 코드를 실행하면 에이전트는 위치 좌표의 갱신 없이 오직 회전 축만을 타겟 방향으로 부드럽게 보간(Interpolate)하여 맞추게 된다.   

심화 디버깅 가이드: 회전 진동(Rotation Wiggling/Jitter) 결함의 수정
루트 모션(Root Motion)을 활용하는 정교한 휴머노이드 캐릭터 애니메이션 환경이나, 공격 모션이 끝날 때마다 특정 방위를 요구하는 상황에서 위 회전 로직을 적용하면, 간혹 에이전트가 완벽히 타겟을 마주보고 섰음에도 불구하고 목적지 위에서 제자리 팽이처럼 끊임없이 미세하게 몸을 떠는 버그가 관찰된다.   

이는 렌더링이나 물리 오류가 아니라, 에셋 내부 스크립트가 "원하는 방향에 정확히 도달했는가?"를 판단하는 관용도(Tolerance) 기준이 비현실적으로 가혹하게 설정되어 있기 때문이다.
내부 로직을 들여다보면, 목표 회전 벡터와 현재 회전 벡터 간의 내적(Dot Product) 값이 임계치(Threshold)를 넘어서야만 회전이 완료된 것으로 간주하는데, JobRepairPath.cs 소스 파일의 280번 라인 근처에 하드코딩된 상수값이 const float ANGLE_THRESHOLD_COS = 0.9999f;로 지정되어 있다. 이는 두 벡터 간의 각도 오차가 소수점 단위 이하일 때만 일치를 인정하겠다는 뜻으로, 부동소수점 오차나 외부 애니메이션 블렌딩이 개입하는 실제 게임 플레이 상에서는 사실상 도달 불가능한 완벽한 0도를 요구하는 셈이다.   

이 떨림 현상을 해소하기 위해서는 해당 스크립트 파일을 열고 상수를 다음과 같이 완화해 주어야 한다.   

C#
// 관용도를 대략 18도 안팎의 오차까지 허용하도록 0.95f로 수정
const float ANGLE_THRESHOLD_COS = 0.95f;
이렇게 임계값을 낮추어 주면 에이전트가 타겟 방향과 대략적으로 일치하는 순간 회전 연산을 종료하고 안정적인 휴식(Idle) 자세를 취하게 되며, 제자리에서 불안정하게 떨거나 빙글 도는 현상을 영구적으로 종식시킬 수 있다.   

6. 역동적 전장 변화의 실시간 대응: Navmesh Cutting 메커니즘
대부분의 3D 전투 게임, 특히 RTS나 공성전 장르에서는 시간이 지남에 따라 지형 자체가 근본적으로 변화한다. 타워가 새롭게 건설되어 길을 가로막거나, 폭격으로 인해 건물 잔해가 무너져 내려 통행로가 폐쇄되는 일은 일상적이다. 이러한 동적인 장애물이 발생할 때마다 앞서 구축한 거대한 Recast Graph 타일을 밑바닥 복셀화 단계부터 전체를 재스캔(Rescan)하는 것은 상상을 초월하는 CPU 병목을 유발하여 게임 전체의 프레임을 심각하게 저하시킨다.   

이 난제를 우회하기 위해 도입된 A* Pro 버전 한정의 최고급 기능이 바로 내비메시 컷(Navmesh Cutting) 시스템이다.   

6.1 절단 알고리즘과 형태적(Shape) 최적화
건물이나 방벽 등 동적으로 전장에 인스턴스화되는 프리팹 구조물 객체에 NavmeshCut 컴포넌트를 부착하면 준비는 끝난다. 이 구조물이 씬에 생성되어 활성화되는 순간, NavmeshCut 시스템은 자신이 속한 구역의 내비메시 타일 데이터를 스캔하는 것이 아니라, 오직 자신의 바운딩 박스(Bounding Box)와 물리적으로 교차하는 기반 트라이앵글(Triangle) 메쉬들만을 찾아내어 빠르고 얕게 해당 부위의 표면 다각형 연산만 도려내어 구멍을 낸다. 타일을 전체 재계산하는 것에 비해 수십 배 이상 빠르다.   

컴포넌트 세부 설정 시, 절취선이 생성될 셰이프(Shape)를 지정하게 되는데 내장된 평면 2D 형태(Rectangle/Circle)보다는 볼륨을 가진 **3D 형태(Cube/Sphere/Capsule)**를 사용하는 것이 압도적으로 권장된다. 3D 게임 환경에서는 경사면이나 단차가 존재하기 마련인데, 2D 셰이프를 사용할 경우 회전값이 주어지더라도 기술적으로는 무조건 수평 방향(XZ 평면)으로만 투영되어 비스듬한 경사로의 구멍을 제대로 절단하지 못하는 논리적 맹점이 존재하기 때문이다. 또한 건물이 파괴되며 크기가 줄어들거나 각도가 뒤틀리는 동적 모션이 수반되는 오브젝트의 경우, 컴포넌트의 useRotationAndScale 필드를 반드시 true로 체크하여 프레임 단위의 외형 변화가 절단 영역에 실시간으로 반영되도록 구성해야 에이전트가 보이지 않는 투명 벽에 가로막히는 오류를 피할 수 있다.   

6.2 Dual Mode 컷팅과 동적 전술 장판(Spell Area)의 융합
내비메시 컷 기술을 단순한 장애물 생성 이상의 차원으로 끌어올리는 기능이 바로 isDual (이중 모드) 체크박스이다.   

일반적인 컷팅은 내비메시에 말 그대로 돌이킬 수 없는 빈 공간의 낭떠러지(구멍)를 뚫어버린다. 반면 isDual 모드를 활성화하면 컴포넌트는 내비메시를 물리적으로 소멸시키는 대신, 경계선을 따라 메쉬를 찢어 분리선(Seam)을 생성하지만 내부의 잘려 나간 조각과 외부의 메쉬를 모두 그대로 유지하여 걷기 가능한 표면으로 남겨둔다.   

이 기법은 마법사가 메테오를 시전하여 바닥이 불타오르는 화염 장판 영역(Fire Spell Area)이나 눈보라가 치는 둔화 영역을 실시간으로 구현할 때 결정적인 역할을 한다.

화염 장판 프리팹이 소환될 때, 부착된 NavmeshCut이 isDual 모드로 해당 원형 영역의 내비메시를 타일로부터 오려내어 독립된 조각으로 분리한다.   

즉시 후속 스크립트나 커스텀 GraphUpdateObject(GUO)를 통해 이 분리된 원형 영역 조각 내부에 앞서 설명한 막대한 '패널티 비용(Penalty)' 값이나 특수 이동 속도 저하 '태그(Tag)'를 덮어씌운다.   

이 일련의 실시간 조작의 결과로, 전장에 위치한 모든 AI들은 화염 장판이 생겨나는 순간 이를 '아예 지나갈 수 없는 완벽한 벽'으로 착각하는 대신, '매우 위험하여 가급적 우회해야 하지만 정 다른 통로가 없다면 대미지를 감수하고서라도 돌파할 수 있는 악조건의 지형'으로 입체적으로 인식하게 된다. 이는 플레이어에게 AI가 매우 정교하게 사고하고 상황을 판단한다는 환상을 불어넣는 궁극의 전투 연출 기법이다.   

다만 주의할 점은, 여러 개의 장판이 겹쳐지거나 거대한 컷팅 영역이 지속적으로 발생하는 환경에서는 잦은 내비메시 절단 업데이트가 Worker Thread의 작업 대기열을 초과시켜 전체 이동 엔진의 반응성을 둔화시킬 우려가 있다는 것이다. 이를 방지하기 위해 전장의 흐름을 총괄하는 게임 매니저 스크립트 최상단에 다음과 같은 스로틀링(Throttling) 방어 코드를 삽입하여 파편화된 다발성 컷팅 요청을 일정 시간 주기마다 하나의 배치(Batch) 작업으로 묶어 처리하도록 제어 간격을 강제해야 한다.   

C#
// 매 프레임 발생하는 미세한 컷팅 요청들을 즉시 처리하지 않고
// 0.1초 동안 모아두었다가 일괄적으로 단 한 번의 업데이트 타이밍에 몰아서 처리함.
AstarPath.active.navmeshUpdates.updateInterval = 0.1f;
특정 시네마틱 컷씬의 종료 시점 등, 지형 변화를 즉시 반영하고 AI들의 경로를 동기화해야 하는 크리티컬한 순간에는 AstarPath.active.FlushGraphUpdates() 명령을 호출하여 백그라운드 스레드의 잔여 업데이트 작업이 완료될 때까지 메인 흐름을 블로킹 대기시킨 후 확정적으로 상태를 갱신할 수 있다.   

결론: 거대하고 복잡한 시스템을 조율하는 단일 통합 해법의 완성
본 보고서 전반에 걸쳐 분석한 바와 같이, 3D 환경에서 다수의 AI가 군집을 이루어 목표를 탐색하고 서로 충돌 없이 교전하는 시스템을 구축하는 과정은 수학적 최적화, 스레드 병렬성 제어, 물리 엔진 충돌 방지, 그리고 동적 맵 데이터 갱신이라는 이질적이고 난해한 공학적 분야들을 아우른다. 이 모든 것을 Unity의 내장 클래스들과 커스텀 스크립트로 직접 구현하여 조합하는 것은 엄청난 유지보수 비용을 요구하며, 결국 프로젝트의 기간 연장과 최적화 실패라는 스파게티 코드의 늪으로 직결된다.   

A* Pathfinding Project 최신 5.x 버전은 이러한 거대한 복잡성을 '데이터 분리' 와 '책임의 철저한 위임' 이라는 아키텍처 철학을 통해 단 하나의 패키지 안에서 완벽하게 통합해냈다.   

개발자는 복잡한 벡터 연산을 직접 타이핑할 필요가 없다.

정적인 지형과 동적인 장애물 관리는 Recast Graph와 NavmeshCut 컴포넌트가 담당하여 안전 구역을 실시간으로 조각해낸다.   

막대한 비용이 소요되는 최단 거리 경로 연산은 Burst 컴파일러와 FollowerEntity 내부에 캡슐화된 ECS 백그라운드 워커가 전담하여 메인 프레임을 방어한다.   

에이전트 간의 엉킴과 충돌은 RVOSimulator 매니저와 RVOController가 비트마스크 피아 식별 논리에 기반한 ORCA 알고리즘을 통해 물 흐르듯 유연하게 비껴가도록 유도한다.   

결과적으로, 이 솔루션을 권장 가이드라인에 따라 Unity 6.3 LTS 시스템에 올바르게 안착시킨다면, AI 프로그래머는 유닛들이 부딪히고 떨리는 하위 레벨의 버그와 씨름하는 대신 "어느 시점에 공격 애니메이션을 재생할 것인가", "적을 인식하면 어떤 우선순위 플래그를 변경할 것인가" 등 상위 단계의 전술적 상태 머신(State Machine) 설계에만 온전히 집중할 수 있게 된다. 사용자 질의에 나타난 "에셋 한 방이면 해결될 것 같다"는 직관은 이 패키지가 제공하는 아키텍처의 설계 의도와 정확히 부합하며, 본 보고서에서 제시한 단계별 통합 방법론을 준수한다면 어떠한 규모의 복잡한 AI 전투 시뮬레이션 환경이든 성공적으로 구현 및 최적화할 수 있을 것이다.
